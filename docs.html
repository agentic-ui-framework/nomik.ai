Getting started

Nomik is an AI workforce you run from chat. For developers, it's also a platform: a REST API to drive workspaces and agents, an MCP tool surface your agents speak natively, and a descriptor model for adding integrations. You build around the same primitives the app uses.

The base URL for the API is https://api.nomik.ai/v1. Everything returns the same envelope, so a response is always success plus data plus metadata (which carries a requestId for support and tracing).

Access is gated to the developer preview today. Request access →

Authentication

Calls are authenticated with a bearer token in the Authorization header. Tokens are signed and verified against a public key set, and carry the principal making the request.

Authorization header
curl https://api.nomik.ai/v1/me \
  -H "Authorization: Bearer $TOKEN"

Coming: scoped, revocable API keys for server-to-server access, so an external app can authenticate as itself rather than as a user session.

API reference

The core resources mirror the product: workspaces, the agents inside them, the runs those agents execute, and the activity they emit. A few representative routes:

Representative routes
GET  /v1/workspaces/:ws/agents          # list agents
POST /v1/workspaces/:ws/agents          # create an agent
POST /v1/agents/:id/runs                # start a run
GET  /v1/workspaces/:ws/activity        # read the activity stream

Response envelope

Success and failure share one shape, so you parse once. Errors carry a machine-readable code alongside a human message.

Error shape
{
  "success": false,
  "error": { "code": "workspace_required",
             "message": "No workspace in context." },
  "metadata": { "requestId": "req_3a9" }
}

MCP & tools

Agents act through tools, and they speak the Model Context Protocol. Every connected integration shows up as MCP tools; so does anything you mount yourself. To extend an agent, you don't learn a bespoke plugin format, you point it at an MCP server.

Mount your own MCP server
{
  "mcpServers": {
    "warehouse": {
      "command": "npx",
      "args": ["-y", "@acme/warehouse-mcp"]
    }
  }
}

Your tools inherit the same risk gating and audit trail as built-in ones: reads run freely, writes that move money or can't be undone wait for approval.

Connectors

A connector is a descriptor, not a service you have to write. Declare the provider's base URL, how it authenticates, and the risk class of its actions; the factory turns that into real, gated tools and stores the credential in the per-workspace vault.

A connector descriptor
export const acme = {
  kind: "acme",
  displayName: "Acme",
  auth: "api_key",         // or "oauth"
  baseURL: "https://api.acme.com/v2",
  risk: "outbound",        // money | destructive | outbound | data | …
}

Risk is the contract: it decides which calls run silently and which pause for a human. Reads on any connector are always free.

Webhooks

Point a provider's webhook at your workspace and inbound events become work. Each delivery is signature-verified, classified by kind, and routed to the squad that handles it.

Inbound webhook endpoint
POST /v1/webhooks/:workspace/:kind/:binding

# verified → classified → published to the fleet
# e.g. a Stripe dispute wakes the finance squad

Coming: outbound event webhooks, subscribe your own URL to fleet events like an approval resolved or a run finished.

Deployment
